## 二、工作流目标

### 2.1 总目标

以后用户在 NX 中选择一个或多个直顶头后，工作流能够完成以下闭环：

```text
选择模型
    ↓
执行审查
    ↓
识别错误规则、错误面、错误实体和具体位置
    ↓
生成结构化问题清单
    ↓
判断问题是否允许自动修复
    ↓
生成修复预览和修复计划
    ↓
执行 NX 几何修复
    ↓
保存修复前后证据
    ↓
重新审查
    ↓
修复成功 / 需要人工确认 / 修复失败回滚
```

### 2.2 第一阶段目标

第一阶段不追求自动修改所有几何，而是先建立可靠的“审查结果—修复计划—复审结果”链路：

1. 审查结果必须能定位到具体实体、规则、面 Tag 和模型位置。
2. AI 必须先解释问题原因，再决定是否可以自动修复。
3. 每次修复必须有修复前快照或可撤销事务。
4. 修复后必须自动重新执行同一条规则审查。
5. 如果复审不通过，必须保留失败记录，不能伪造为修复成功。
6. 无法可靠定位或修改的几何问题必须转为“人工确认”，不能盲目改模。

### 2.3 非目标

第一阶段暂不包含以下能力：

- 不自动改变模具设计意图。
- 不自动修改 NX 安装目录或 NX 系统文件。
- 不自动删除用户模型中的特征、实体或历史记录。
- 不自动修复无法明确定位的自由曲面、复杂倒扣和拓扑损坏。
- 不把“重新审查通过”当作“设计一定符合全部行业标准”。
- 不在没有用户明确触发的情况下自动检查 PRT 模型。

## 三、当前冻结基线

### 3.1 冻结原则

当前通过人工验收的 DLL、Python、DLX 和源码作为只读基线。后续任何自动修复开发必须在冻结版本之外进行，不能直接覆盖冻结目录中的文件。

冻结目录中保留：

- `src`：当前源代码和工程文件。
- `tools`：当前可以在 NX 中运行的 DLL、Python、DLX 和运行脚本。
- `Command`：当前开发规划、同步规则和界面预览。
- `冻结清单.md`：冻结文件列表。
- `SHA256SUMS.txt`：冻结文件 SHA-256 校验值。

### 3.2 当前已确认审查规则

| 规则 | 审查标准 | 失败反馈 |
|---|---|---|
| R1 | 四个主侧面相对 Z 轴的角度小于 `3°` 时不合格 | 对应错误侧面标红 |
| R2 | 直顶头 Z 向高度低于 `30 mm` 时不合格 | 弹出“直顶高度不够，无法正常放下直顶杆”，并提示还需增加多少高度，逐个定位到错误实体 |
| R3 | 底面与 Z 轴应为 `90° ± 0.1°` | 只标红实际错误底面，不能误标侧面 |
| R4 | 先按实体中心校正向外法向；相对 `+Z` 的有符号拔模角大于 `0°` 时不合格 | 标红对应反向拔模面 |
| R9 | 用户输入最小圆角半径；输入为 `0 mm` 时关闭 | 小于输入值的可识别圆角面标红，并询问用户是否需要调整圆角 |

### 3.3 当前已确认交互

- 支持一次选择多个实体。
- 按实体逐个审查并汇总结果。
- 高度错误实体逐个弹窗、逐个定位，并提示建议补高数值。
- 定位过程采用移动视图，不直接瞬移到最终位置。
- 错误面在用户点击确定后保持红色。
- 命令结束后支持清除本次审查高亮。
- UI 只保留实体选择和最小圆角输入；其他规则使用默认值。
- DLL 与 Python 必须同步修改、同步测试、同步复制到 `tools`。

## 四、自动修复总体架构

### 4.1 五层结构

#### 第一层：NX 输入层

负责从 NX 获取：

- 当前选择的实体。
- 实体 Tag、名称和所属 Part。
- 面、边、顶点的 Tag。
- WCS 和 Z 轴方向。
- 面的类型、法向、位置、面积、半径等几何信息。
- 当前视图和定位所需的包围盒。

输入层只负责读取和创建 NX 事务，不负责自行改变设计规则。

#### 第二层：审查层

调用当前已经通过验证的 R1、R2、R3、R4、R9 规则，生成统一格式的审查结果。审查层必须区分：

- 通过。
- 明确失败。
- 无法可靠识别。
- 需要人工确认。

不能把“无法识别”直接当成“通过”。

#### 第三层：问题理解层

将审查结果转换成 AI 可理解的问题对象，至少包含：

- `issue_id`：问题编号。
- `body_tag`：错误实体 Tag。
- `face_tags`：相关错误面 Tag 列表。
- `rule_id`：R1、R2、R3、R4 或 R9。
- `measured_value`：实测值。
- `required_value`：设计标准值。
- `suggested_delta`：建议调整量，R2 时表示还需增加多少高度，R9 时表示建议调整的圆角差值。
- `reason`：判错原因。
- `location`：模型定位信息。
- `repairability`：可自动修复、需人工确认或禁止自动修复。
- `risk_level`：低、中、高。

#### 第四层：修复执行层

根据问题类型调用明确的 NX 修复动作。修复动作必须具备：

- 明确目标实体或面 Tag。
- 明确修改参数和单位。
- 明确修改前状态。
- 可撤销事务或修复前副本。
- 执行后的结果返回值。
- 修复后重新审查入口。

#### 第五层：证据与发布层

负责保存：

- 修复前审查结果。
- AI 修复计划。
- 实际执行的 NX 操作。
- 修复后审查结果。
- 失败原因或人工待确认项。
- 版本号、文件哈希和操作时间。

## 五、AI 自动修复决策规则

### 5.1 自动修复前必须满足的条件

AI 只有在以下条件全部满足时才能自动修改：

1. 错误规则已经明确。
2. 错误实体 Tag 已明确。
3. 错误面 Tag 已明确，或规则只涉及实体整体参数。
4. 修复动作在 NX API 中有确定实现。
5. 修改参数有明确单位和范围。
6. 修复不会依赖 AI 猜测设计意图。
7. 修改前能够创建 NX Undo Mark 或等价回滚点。
8. 修复后可以重新执行对应审查规则。

### 5.2 问题分级

| 等级 | 含义 | AI 行为 |
|---|---|---|
| A 类 | 目标和修改动作完全明确，风险低 | 可直接执行，执行后必须复审 |
| B 类 | 可以提出修复方案，但存在尺寸、方向或设计意图不确定 | 生成方案并定位，等待人工确认后执行 |
| C 类 | 无法可靠识别目标或修复动作可能破坏模型 | 禁止自动修改，只输出问题和人工处理建议 |

### 5.3 当前规则的初始修复边界

| 规则 | 第一阶段自动修复策略 |
|---|---|
| R1 | 可定位角度不足的明确平面时，生成拔模方向/角度修复方案；具体修改方式先列为 B 类，不能直接猜测基准面和设计尺寸 |
| R2 | 仅能可靠报告高度不足并定位实体；同时计算建议补高量，高度补偿涉及顶面、底面或特征历史，第一阶段列为 B 类 |
| R3 | 能明确定位错误底面时，可生成底面方向修复方案；涉及实体重建时列为 B 类 |
| R4 | 能明确定位正拔模面时，可生成反向拔模修复方案；该规则第一版直接进入自动修复执行队列，必须保留实体中心校正后的法向和角度证据 |
| R9 | 能识别解析圆角且目标圆角面明确时，可生成圆角半径调整方案，并在执行前询问用户是否需要调整圆角；自由曲面、倒角和非解析面列为 C 类 |

> 第一阶段的重点不是强行自动改几何，而是先建立安全的修复决策和复审闭环。只有经过专用错误 PRT 验证的修复动作，才能从 B 类升级到 A 类。

## 六、单次工作流执行步骤

### 步骤 0：用户触发

用户在 NX 中打开模型，选择一个或多个直顶头，明确触发“审查”或“审查并修复”。未收到触发时，插件不自动读取或修改 PRT。

### 步骤 1：创建运行批次

生成唯一的 `run_id`，记录：

- 当前 Part 路径和名称。
- NX 会话时间。
- 当前插件版本。
- 当前 WCS Z 轴。
- 用户选择的实体 Tag 列表。
- 最小圆角输入值。
- R2 预估补高量。
- R9 调整确认结果。

### 步骤 2：执行只读审查

先执行完整审查，不允许在第一轮审查过程中修改实体。输出每个实体的规则状态、实测值、错误面和定位数据。

### 步骤 3：生成问题清单

将所有失败项按实体分组，再按规则排序。每个问题必须包含“为什么错”和“错在哪里”，避免只输出“检查失败”。

### 步骤 4：判断修复权限

对每个问题执行自动修复决策：

- A 类问题进入自动修复队列。
- B 类问题进入待确认队列并给出可视化定位。
- C 类问题进入人工处理队列。

### 步骤 5：生成修复计划

修复计划必须先写入结果文件，再执行 NX 修改。计划至少包括：

- 目标实体和面。
- 原始实测值。
- 目标标准值。
- 建议修改量。
- 预计影响的特征。
- 风险等级。
- 是否需要人工确认。

### 步骤 6：建立回滚点

执行任何修改前：

1. 创建 NX Undo Mark。
2. 记录目标对象的 Tag 和关键几何数据。
3. 写入修复前结果。
4. 如果工作流支持文件级备份，则在用户指定位置生成备份，不覆盖原始 PRT。

### 步骤 7：执行修复

每次只处理一个明确问题，处理完成后立即返回：

- 执行成功或失败。
- 实际调用的操作。
- 实际修改的对象。
- 实际使用的参数。
- NX 返回的错误信息。

多个问题不能在没有中间记录的情况下合并成一个不可追踪操作。

### 步骤 8：修复后复审

对每个已修复问题重新执行原规则，并至少检查：

- 原错误是否消失。
- 是否产生新的错误面。
- 是否影响同一实体的其他规则。
- 是否影响同一批次中的其他实体。
- 模型是否仍然有效、可显示、可保存。

### 步骤 9：确定结果

结果分为：

- `修复成功`：原问题消失，相关回归规则通过。
- `部分修复`：部分问题消失，仍有问题保留。
- `修复失败`：修改未生效或复审仍失败。
- `需人工确认`：方案存在但不能安全自动执行。
- `已回滚`：修复造成异常，已恢复到修复前状态。

### 步骤 10：输出并保留高亮

- 错误面继续保持红色，直到用户点击清除高亮或重新开始审查。
- 修复成功的面可以恢复原显示，也可以在结果中保留修复标记；具体显示方式在 Block UI 验收时固定。
- 高度错误继续逐个定位和弹窗，不把多个错误压缩成一个无法对应的提示；每个高度错误必须输出“还需增加多少高度”。

## 九、DLL 与 Python 同步要求

### 9.1 必须同步的内容

- R1、R2、R3、R4、R9 的判断条件。
- 面和实体 Tag 的获取方式。
- 错误面标红和清除高亮行为。
- 多实体执行顺序。
- 高度错误定位和窗口文本。
- 修复计划字段和复审结果字段。
- 异常处理和回滚结果。

### 9.2 每次修改后的顺序

1. 先更新开发文档和结果协议。
2. 修改 `src` 中 DLL 与 Python 的对应逻辑。
3. 编译 DLL，并确认输出是当前版本。
4. 同步复制 DLL、Python、DLX 和必要脚本到 `tools`。
5. 运行静态检查和结构化测试。
6. 在 NX 中分别验证 DLL 和 Python。
7. 记录人工反馈和截图。
8. 只有通过复审后，才创建新的冻结版本。

禁止只修改 `tools` 中的运行文件而不更新 `src`，也禁止 DLL 和 Python 使用不同的审查标准。

## 十、安全、回滚与权限边界

### 10.1 修改安全

- 默认先审查、后计划、再修改。
- 任何修改前必须建立 Undo Mark 或备份。
- 每次只修改一个可追踪问题。
- 修改目标必须使用明确的实体或面 Tag，不能通过“第几个面”猜测。
- 不允许通过删除实体来掩盖审查失败。
- 不允许用固定成功返回替代真实 NX 结果。
- 不允许修改 NX 安装目录。

### 10.2 回滚条件

出现以下任一情况时立即停止当前批次并回滚：

- NX API 返回错误。
- 修复后实体丢失、不可显示或拓扑数量异常。
- 修复后原规则仍失败且没有可解释的变化。
- 新增了未预期的错误面。
- 无法确认当前模型状态是否与修复动作对应。
- 修复操作影响了未选中的实体。

### 10.3 外部 AI 的权限边界

外部 AI 可以：

- 读取结构化审查结果。
- 解释规则失败原因。
- 生成修复计划。
- 根据已批准的动作调用修复执行器。
- 读取修复后复审结果并生成报告。

外部 AI 不可以：

- 自行改变规则阈值。
- 自行选择未报告的实体或面。
- 自行删除或替换整个模型。
- 在没有修复计划和回滚点时执行修改。
- 把人工确认项伪装成自动修复成功。

## 十一、测试与验收路线

### 11.1 自动测试层

首先验证不依赖 NX 图形界面的内容：

- JSON 字段完整性。
- R1/R2/R3/R4/R9 状态与数值。
- 问题分级是否正确。
- 修复计划是否包含目标 Tag 和参数。
- 失败时是否保留错误信息。
- 回滚状态是否正确记录。
- 修复前后结果是否能对应同一 `issue_id`。

### 11.2 NX 人工验收层

必须在 NX 中验证：

1. 多实体选择是否仍然正常。
2. 错误面是否标红且位置准确。
3. 高度错误是否逐个弹窗和定位。
4. 视图移动过程是否平滑，最终大小符合当前约定。
5. 点击确定后错误面是否保持红色。
6. 清除高亮后是否恢复显示。
7. 修复动作是否只影响目标实体。
8. 修复后重新审查是否显示正确结果。
9. 取消、异常和回滚后模型是否保持原状。

### 11.3 必备测试模型

| 类型 | 用途 |
|---|---|
| 正确直顶头 | 验证无误报和修复流程不应修改合格模型 |
| R1 错误模型 | 验证侧面角度不足和面定位 |
| R2 高度不足模型 | 验证高度报告、逐个定位和人工修复方案 |
| R3 底面错误模型 | 验证底面识别不能误标侧面 |
| R4 反向拔模模型 | 验证有符号拔模角和错误面定位 |
| R9 小圆角模型 | 验证输入阈值、关闭状态和圆角面定位 |
| 多实体混合模型 | 验证一个批次内多个实体和多个错误同时存在 |
| 无法识别模型 | 验证 C 类问题不会被强行修改 |

### 11.4 通过标准

只有同时满足以下条件，才允许创建新的工作流冻结版本：

- 自动测试通过。
- DLL 与 Python 结果一致。
- 修复前后结果可追踪。
- 修复失败可以回滚。
- NX 人工验收通过。
- 无未解释的误标红、漏检或模型损坏。
- 运行日志、测试结果和人工反馈已归档。

## 十二、工作流状态机

```text
IDLE
  ↓ 用户触发
COLLECTING_INPUT
  ↓ 选择和参数校验通过
REVIEWING
  ↓
ISSUES_FOUND ─────────────→ NO_ISSUE
  ↓
PLANNING_REPAIR
  ├── C 类 / B 类 → WAITING_HUMAN_CONFIRM
  └── A 类       → READY_TO_REPAIR
                        ↓
                  CREATING_UNDO_MARK
                        ↓
                  APPLYING_REPAIR
                   ├── 失败 → ROLLING_BACK
                   └── 成功 → VERIFYING
                                  ├── 通过 → REPAIR_SUCCESS
                                  ├── 部分通过 → PARTIAL_SUCCESS
                                  └── 失败 → ROLLING_BACK 或 HUMAN_REVIEW
```

每个状态都必须写入日志，不能只保留最终一句“完成”。

## 十三、推荐实施顺序

### 阶段 1：协议先行

1. 固定审查结果 JSON 格式。
2. 固定问题对象字段。
3. 固定修复计划字段。
4. 固定成功、失败、回滚和人工确认状态。

### 阶段 2：只读工作流

1. 调用当前冻结插件执行审查。
2. 读取结果并生成问题清单。
3. 自动定位错误实体和错误面。
4. 生成 Markdown 报告。
5. 暂不修改模型。

### 阶段 3：修复计划工作流

1. 为 R1、R2、R3、R4、R9 建立修复能力矩阵。
2. 将问题分为 A、B、C 三类。
3. 只对 A 类生成可执行计划。
4. B 类只生成方案并等待确认。
5. C 类只生成解释和人工处理建议。

### 阶段 4：单规则修复

按 R1、R3、R4、R9、R2 的顺序逐条实现，每实现一条就完成：

```text
专用错误模型
    ↓
修复动作
    ↓
Undo / 备份
    ↓
修复后复审
    ↓
NX 人工验收
```

R2 高度不足通常会涉及实体尺寸、顶面、底面或特征历史，优先做“定位 + 方案”，最后再决定是否自动修改。

### 阶段 5：多实体与异常闭环

1. 支持多个实体分别生成修复任务。
2. 一个任务失败时不影响其他任务的日志记录。
3. 明确单体回滚和整批次回滚策略。
4. 修复后统一复审并输出汇总。

### 阶段 6：新版本冻结

工作流达到验收标准后，在 `Command\冻结版本` 下创建新的版本目录，保留源代码、运行文件、配置、测试报告、日志和 SHA-256 清单。旧冻结版本永远不覆盖。

## 十四、当前不确定项与处理原则

| 项目 | 当前状态 | 处理方式 |
|---|---|---|
| R1 的具体几何改法 | 待专用错误模型验证 | 先生成修复方案，不直接改模 |
| R2 高度由哪一侧补偿 | 待设计意图确认 | 只定位和报告，列为 B 类 |
| R3 是否允许重建底面 | 待模型历史确认 | 优先保留原特征结构 |
| R4 修复基准面 | 已确认 | 必须使用明确基准，按实体中心校正后的法向和目标面 Tag 执行；无需人工确认 |
| R9 自由曲面圆角 | 不可靠 | 列为 C 类，不自动修改 |
| 自动保存修复后 PRT | 待确认 | 默认不覆盖原文件，保存前生成备份 |
| B 类问题是否自动弹确认窗口 | 待 UI 设计 | 第一版仅对 R9 圆角确认采用弹窗，R4 不再进入人工确认 |

不确定项必须保留在报告中，不能通过默认值或猜测消除。
