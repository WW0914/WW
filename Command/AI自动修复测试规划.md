## 二、测试目标

测试必须证明以下事实：

1. 当前审查插件的 R1、R2、R3、R4、R9 结果没有因工作流集成发生退化。
2. DLL 与 Python 对同一模型输出一致。
3. 审查结果能够稳定转换成统一问题对象。
4. AI 只能为明确问题生成修复计划。
5. 没有注册修复动作时不会修改模型。
6. 修改前一定存在 Undo Mark 或备份。
7. 修复后一定重新执行完整审查。
8. 修复失败、出现新问题或模型失效时能够回滚。
9. 多实体场景能够逐体定位、记录、修复和复审。
10. 所有结果可通过 `run_id`、`issue_id`、`plan_id` 追踪。

## 三、测试层级

### 3.1 L0 文档与静态检查

- 文件存在性。
- UTF-8 编码。
- JSON 示例语法。
- 配置枚举与结果协议枚举一致。
- 路径和版本号引用一致。
- 冻结目录未被修改。

### 3.2 L1 纯逻辑单元测试

不启动 NX，测试：

- 配置解析和默认值。
- 阈值边界比较。
- 结果对象序列化。
- 原因码映射。
- A/B/C 分类。
- 修复计划生成。
- 状态机转换。
- JSON Schema 校验。

### 3.3 L2 NX 只读集成测试

启动 NX 并打开测试 PRT，但不修改模型，测试：

- 实体与面 Tag 读取。
- R1/R2/R3/R4/R9 结果。
- 多实体选择顺序。
- 标红、定位、窗口和清除高亮。
- `review-before.json` 输出。

### 3.4 L3 NX 修复执行测试

只对已经批准的单规则修复动作执行：

- Undo Mark。
- 目标对象锁定。
- 几何修改。
- 执行记录。
- 修复后完整复审。
- 回滚。

### 3.5 L4 Block UI 人工验收

用户按真实操作顺序验证：

- 选择行为。
- 参数输入。
- Apply/OK/Cancel。
- 弹窗文本。
- 视图移动。
- 红色保持与清除。
- 修复确认和保存行为。

自动测试通过不能代替 L4 人工验收。

```text
L0 文档与静态检查
    ↓

    ↓
L2 NX 只读审查测试
    ↓

    ↓
单个规则修复动作的专用测试
    ↓

    ↓

    ↓
Block UI 人工验收
    ↓
生成报告与新版本冻结
```

如果前一层失败，不进入后一层模型修改测试。

## 二十一、缺陷分级

| 等级 | 定义 | 示例 | 处理 |
|---|---|---|---|
| P0 | 可能损坏模型、错误保存或无法回滚 | 覆盖原 PRT、修改错实体 | 立即停止 |
| P1 | 审查或修复结论严重错误 | R3 标红侧面、R4 方向反判 | 修复后重新全量回归 |
| P2 | 功能可用但交互或记录不完整 | 定位大小不合适、日志缺字段 | 当前版本修复 |
| P3 | 文档或显示小问题 | 非关键措辞、排版 | 记录后处理 |

存在 P0/P1 时禁止冻结为自动修复可用版本。

## 二十二、准入门禁

### 门禁 A：只读工作流通过

- 配置加载和安全默认值通过。
- 结果协议 Schema 通过。
- R1/R2/R3/R4/R9 输出完整。
- DLL 与 Python 结果一致。
- 不修改模型。

### 门禁 B：修复计划通过

- A/B/C 分类准确。
- 没有动作时不会生成可执行计划。
- 目标 Tag、参数、单位、风险和前置条件完整。
- 人工确认项不会被自动执行。

### 门禁 C：单规则修复通过

- 专用错误模型通过。
- 修复前有 Undo Mark 或备份。
- 修复后完整复审通过。
- 失败和新增问题可回滚。
- NX 人工验收通过。

### 门禁 D：多实体工作流通过

- 多实体顺序正确。
- 单个问题可独立追踪。
- 单体失败不丢失其他结果。
- 模型失效时整批保护有效。

### 门禁 E：版本冻结

- P0/P1 为 0。
- 自动测试和人工验收证据齐全。
- DLL、Python、DLX、配置和协议一致。
- 生成冻结清单和 SHA-256。
- 不覆盖旧冻结版本。
